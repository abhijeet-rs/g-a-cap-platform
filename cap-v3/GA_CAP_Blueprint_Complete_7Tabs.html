Three Data Classes (Foundation PRD Principle)

Every field in the platform falls into one of three classes with distinct edit rules F1

Synced (Prism = SoR)

Read-only in the platform. Pulled from Prism; never editable here (editing would fork the SoR).

Examples: Master plans, carrier base rates, plan-code crosswalks, client records (for renewals).

G&A-owned (Platform = SoR)

Editable centrally by admins; versioned with effective dates. Changes propagate to draft CAPs; signed CAPs keep their snapshot.

Examples: Pricing-stack factors, parameters, fees, templates, vocabularies, validation rules, roles.

Connected (External Systems)

Configured here (credentials, mappings); data flows via integration. The platform orchestrates but does not own the data.

Examples: ClientSpace (workflow/cases), DocuSign (e-sign).

Master Plan vs. Open Market — Where it Matters in Architecture

A single client carries both types simultaneously. The cloud tool must fork behavior based on plan type selection in the Products Grid. Here's where each architectural component behaves differently:

Master Plan path

Rate source: F3 carrier base rate × F4 risk factor F3F4
Plan catalog: G&A master plan codes with validated rates
Validation: F8 source-data rules check codes against Prism catalog
Renewal pre-fill: R2 full rate recalculation with new UW params

System Integrations (Overview PRD §5)

The governing rule: clean orchestration layer

The CAP platform is a clean orchestration layer on top of Prism — not a rip-and-replace, and not a second system of record. It reads master/rate/client data from Prism, owns the quoting/modeling/approval layer Prism doesn't model, and writes back the final config. All external system access routes through the F2 Connections Hub.

PrismHR / WorkSight F2 F3

System of Record for plan configs, carrier rates, enrollments, payroll. Two sync streams (master data + client records). Read: plans, rates, census. Write-back: NB9 (create) / R9 (update) via BenefitService. WorkSight = client-facing enrollment portal.

ClientSpacePRO F2

DocuSign F2

Build an abstraction layer (Connections Hub, F2) that normalizes all external system interactions. Each integration is a module with its own auth, retry, and error handling. SOX Snapshot-on-Finalize (Overview PRD §6.4)

Versioning + snapshot-on-finalize — immutability for audit/SOX

All G&A-owned data is effective-dated and versioned. The critical rule:

  • A draft CAP reads the current live version of foundation data (so central edits propagate forward to in-progress CAPs).
  • A finalized/signed CAP snapshots the exact versions it used and locks them (immutability for audit/SOX). The signed booklet is an immutable artifact pinned to its data snapshot.
  • This is what makes "edit once, flows to all future CAPs" safe — signed CAPs are never retroactively affected by central changes.
  • The prior signed booklet stays immutable on its snapshot; the renewal becomes a new versioned artifact on current data, yielding a clean year-over-year trail of which data each approval used.

Applies to: NB7NB8 booklet sign-off, R7R8 annual contract management, F3F4F5 foundation versioning.

Copilot PII/PHI Minimization CP7

Guardrails for the Copilot agent in a PHI/PII/SOX context

  • PII/PHI minimization: Send structured references and the minimum necessary fields to the LLM, not raw census/SSN dumps. Redact/limit sensitive payloads per data-governance rules. Document what is and isn't sent to the model.
  • Confirmation on all writes (CP4): No silent mutation. Every change shown as before-after preview, user confirms before apply. Explicit refusal + reason when an action isn't permitted.
  • RBAC parity (F9): Agent action set = platform capabilities intersected with current user's role permissions. Cannot exceed the user's own rights. Cannot edit synced/read-only fields. Cannot modify signed/locked artifacts.
  • Grounding / anti-hallucination: Factual claims must be backed by a retrieval result. If unavailable, the agent states it can't find the data rather than inventing a rate, factor, or field value.
  • Full audit: Every read (optional for sensitive data) and every write logged with actor, timestamp, tool, inputs, before-after, and confirmation event. Logged as kind: "ai" in the AuditLog.
  • Source transparency: Every surfaced fact and proposed change shows where it came from and (for rates/factors) which version.
What Data the CAP Platform Handles

Data scope expands beyond the Excel file

The CAP Excel itself is primarily plan-level financial data — rates, contribution strategies, underwriting parameters, broker info. It does NOT contain individual SSNs, health claims, or dependent birthdates. But the moment we integrate with PrismHR via API, the scope expands: getActiveBenefitPlans, getDependents, getEmployeePremium, and enrollBenefit all touch employee-level data that IS PHI and PII. The platform must be architected for the full scope, not just the CAP document.

What's IN the CAP data

  • Company identity: Client name, Prism ID, corp type, affiliate IDs
  • Underwriting financials: Bucket, admin factor, commission %, risk factor, renewal increase
  • Plan rates: Total/ER/EE per plan per tier per class (dollar amounts)
  • Contribution strategies: %, fixed $, variable — how employer vs. employee costs are split
  • Broker compensation: Commission values, flat $ amounts, "Outside Broker paid" flags
  • Personnel names: Client contact (Mark Kirby), AM (Danielle Andre), BS (Nataly Montano), Broker company (Digital Insurance LLC)
  • FSA/HSA parameters: ER contribution amounts per tier, deposit frequency, proration rules
  • Admin config: Enrollment dates, billing methods, COBRA status

What's accessed via Prism integration

  • Employee demographics: Names, SSNs, DOB, addresses, pay rates (via EmployeeService)
  • Dependent information: Names, DOB, relationship, SSNs (via BenefitService.getDependents)
  • Enrollment selections: Which plan each employee chose, coverage tier, effective dates (via getActiveBenefitPlans)
  • Premium billing: Employee-level billing rates tied to specific individuals (via getEmployeePremium)
  • Deduction data: Payroll deduction amounts per employee (via PayrollService/DeductionService)
  • Life events: Status changes, qualifying events, COBRA eligibility (via EmployeeService.addEmployeeEvents)

This data is PHI + PII. The platform is subject to HIPAA, even though the CAP document itself is mostly plan-level.

Data Classification Scheme

Field-level classification — not all CAP data is equally sensitive

Every field in the data model must be tagged with a sensitivity level that drives access control, encryption, logging, and retention behavior:

Public / Internal

Plan names, carrier names, plan design parameters (deductible, copay, coinsurance, OOP max), enrollment method, packet type, language. No access restriction beyond authenticated user.

Example: "BCBSTX PPO $500 80%" is a plan name — not sensitive.

Confidential

Employer contribution strategies, commission percentages, broker compensation, underwriting factors (bucket, AF, RF), renewal increase %, rate relief flags. Business-sensitive and competitive.

Example: Itafos bucket 0.975 — competitors or other clients must never see this.

PHI / Restricted

Any data that, combined with employee identity, constitutes protected health information: enrollment selections, coverage tier, plan effective dates, premium amounts tied to individuals, dependent enrollment.

Example: "Employee #4521 enrolled in HDHP $3300 with spouse" — this is PHI under HIPAA.

PII

SSNs, employee names + DOB, dependent names + DOB, addresses, wage data, client contact personal info, broker contact info. Subject to state privacy laws (CCPA/CPRA, TDPSA).

Example: SSN from Prism EmployeeService — must never be stored in the CAP tool.

Compliance Framework

HIPAA Security Rule

G&A Partners is a covered entity (administers health benefits for ~130K WSEs). Any system that stores, processes, or transmits PHI must comply:

  • Administrative safeguards: Security officer designation, workforce training, access authorization procedures, incident response plan
  • Physical safeguards: Cloud provider must have SOC 2 Type II and HIPAA BAA (AWS, Azure, GCP all offer this)
  • Technical safeguards: Encryption at rest (AES-256) and in transit (TLS 1.2+), unique user IDs, automatic session timeout, audit logging of all PHI access, emergency access procedures
  • BAA required: Business Associate Agreement between G&A and the cloud provider, and between G&A and R Systems as the platform developer

SOX-Adjacent Controls

The PRD explicitly flags "SOX-adjacent risk" for carrier enrollment transcription errors. The CAP tool handles financial data that flows into payroll deductions:

  • Audit trail: Every change to rates, contributions, and plan configurations logged with who, when, what, old value, new value — not optional, it's a control requirement
  • Segregation of duties: AM creates → Coordinator reviews → Client signs → Analyst configures. No single person can create AND approve a CAP NB6 R6
  • Reconciliation support: Platform must provide canonical rate data that reconciliation processes (Init F / Beacon) can verify against — the CAP rates ARE the control point for payroll-vs-carrier invoice matching
  • Change control: Plan configuration changes (new carriers, new plan codes, rate table updates) must go through an approval workflow, not ad-hoc edits F3 F4
  • Snapshot-on-finalize: Signed CAPs lock their data versions. Immutable signed booklets pinned to their data snapshots. Year-over-year audit trail via R7 annual contract management.

State Privacy Laws

G&A operates across multiple states. Client employees may be in jurisdictions with specific privacy requirements:

  • CCPA/CPRA (California): Right to know, right to delete, right to opt-out. Applies to employee PII if G&A has California-based client employees. The CAP tool must support data subject requests.
  • TDPSA (Texas): G&A is headquartered in Texas. Texas Data Privacy and Security Act requirements apply.
  • Other states: Colorado, Virginia, Connecticut, Utah, and others have enacted privacy laws. The platform should support configurable privacy controls per jurisdiction.

Note: Most state privacy laws exempt data already covered by HIPAA. But employee data beyond health coverage (contact info, wage data) may still be subject.

ERISA Record Retention

Employee Retirement Income Security Act requires retention of employee benefit plan records:

  • Plan documents: Must be retained for duration of the plan + 6 years after termination
  • Enrollment records: 6 years minimum after the plan year ends
  • Financial records: 6 years for contribution and premium records
  • CAP implication: Historical CAPs must be retained for at least 7 years (plan year + 6) to satisfy ERISA

MCP Server Architecture — AI-Native Integration Layer

What is MCP and Why It Matters Here

Model Context Protocol (MCP) in 30 seconds

MCP is an open standard (by Anthropic) that lets AI models call external tools through a simple, typed interface. Think of it as a USB-C port for AI — any AI agent that speaks MCP can plug into any MCP server, without custom integration code per model.

  • MCP Server = a thin wrapper around an API/database/service that exposes typed tools (functions the AI can call) and resources (data the AI can read)
  • MCP Client = the AI agent (e.g., CAP Copilot, Claude, a custom agent) that discovers and calls those tools
  • Transport = stdio (local) or SSE/HTTP (remote) — the server can run anywhere

For G&A: instead of hard-coding PrismHR API calls into the Copilot (CP3/CP4), we wrap each external system as an MCP server. The Copilot then discovers available tools at runtime, calls them with typed parameters, and gets structured results — all with built-in permission checking, rate limiting, and audit logging.

Without MCP (Current approach)

Every integration is hand-wired into the app. Adding a new data source means modifying Copilot code. Each AI feature needs custom fetch/parse/validate logic. No standardization across integrations.

With MCP Servers

Each system is a standalone MCP server. Any AI agent can discover and call tools. Adding a new data source = deploying a new MCP server. Copilot, Claude, or future agents all use the same tools. Audit logging at the MCP layer, not per-feature.

Security & Guardrails for MCP Servers

Authentication & Authorization

  • Each MCP server authenticates to its backend with service accounts — not user credentials
  • The MCP layer passes the user context (role, permissions from F9) with every call
  • WRITE tools check user.permissions.includes('edit') before executing
  • Synced/read-only fields (Prism data) refuse writes at the MCP level — the AI never even gets a write tool for those
  • PII/PHI fields are redacted or summarized in MCP responses — e.g., census returns counts, not SSNs

Audit & Compliance

  • Every MCP tool call is logged: { timestamp, user, tool, params, result_summary, duration_ms }
  • Write tool calls additionally log: { before, after, confirmed_by, confirmation_method }
  • Logs feed into the platform's AuditLog entity (Tab 6) and the ClientSpace audit trail
  • SOX compliance: signed/finalized CAPs reject ALL write tool calls at the MCP layer
  • Rate limiting: max 100 calls/minute per agent session, with backoff
