PrismHR Integration Context — what we're building and why
What is PrismHR?
G&A's primary HRIS and payroll processing platform. PM creates the client shell (locations, positions, departments); PA configures billing rules, SUTA rates, deduction codes, pay groups, and runs mock and initial payrolls.
What is Being Integrated?
Pre-flight validation engine reads PrismHR configuration data via REST API, validates against CSA Schedule 2 (source of truth), and writes fixes back via PrismHR PUT endpoints — eliminating manual navigation.
Why Pre-Flight?
Today, configuration errors (WC codes, SUTA, billing) are only discovered when mock payroll fails. Pre-flight catches them before initialization, replacing the iterative calculate-fail-fix cycle.

SUTA Pre-Approval
Pre-approval happens during the sales process. The PA selects rate type — not triggers approval. This removes the executive queue from the critical path for most deals:
Standard Rates — Pre-Approved at Deal Close
Auto-approved during sales for common states: TX, CA, FL, NY, IL, PA, OH, GA. PA selects "Standard Rate" and it is already approved — no executive review needed.
Non-Standard / Client Rate (CR%)
For states like CT where the CSA specifies Client Rate, the PA enters the rate and it is escalated to executive review with 1-week SLA.
Removes approval queue from critical path for ~80% of deals — only exception cases require executive sign-off.

Validation Rule Engine
Deterministic
Rule-based, deterministic — no AI/ML needed. 5 rules execute sequentially, each with clear pass/fail criteria:
WC Code: JOIN employee roster WC codes against CSA Schedule 2 extracted codes — flag any employee whose code is not in the CSA set
SUTA: Cross-reference PrismHR SUTA rate table against distinct employee states — flag any state with active employees but no rate entry
Billing: Compare PrismHR admin fee / PEPM config against CSA extracted fee structure — flag amount mismatches
Pay Group: Verify pay group frequency matches CSA billing frequency (semi-monthly, bi-weekly, etc.)
Deductions: Validate each active deduction code references a configured GL account in PrismHR
Pass/Fail Output & Remediation
Per-check: status (PASS/FAIL), severity (Critical/Warning/Info), affected count
Critical failures: block mock payroll initialization until resolved
Each failure includes: what’s wrong, which employees/codes affected, exact fix steps
Report format: on-screen interactive + PDF export for audit trail
PA fixes issues inline via dropdowns and inputs — all changes pushed to PrismHR in one batch API call. No context-switching required.
SUTA Executive Pre-Approval Logic
Pre-approval happens during the sales process. The PA does not trigger pre-approval — they select rate type. If Standard, it is already approved. If Client Rate, it gets escalated.
Standard Rates (Pre-Approved)
TX, CA, FL, NY, IL, PA, OH, GA — common states where standard rates are auto-approved at deal close during sales. No executive review needed.
Non-Standard / Client Rate (CR%)
For states like CT where CSA specifies Client Rate, PA enters the rate and it is escalated to executive review with 1-week SLA.
Integration: During CSA extraction, SUTA coverage type is identified (CR%, Standard, Full). If Standard → pre-approved rate applied, PA selects and proceeds. If CR% → PA enters client-specific rate → auto-escalate to executive queue with 1-week SLA.
Removes the executive approval bottleneck from the critical path for ~80% of deals — only exception cases require executive sign-off
